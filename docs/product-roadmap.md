# 医美机构账期评估系统产品化开发路线图

## 当前判断

这个项目需要产品开发路线图。

当前实现已经完成了一个可交互 H5 原型，核心价值是验证业务规则、移动端表单结构和结果表达。但如果目标是“可上线、可复用、可沉淀数据、可进入内部业务流程”，就不能继续按 demo 思路堆功能。下一阶段必须转为小步 PR、明确数据边界、明确上线环境、明确验收口径。

## 产品目标

把“医美机构账期评估系统”从本地 H5 原型推进为内部可用的授信评估工具：

- 业务人员可用手机访问并填写评估。
- 系统实时输出授信建议、等级、账期、额度和原因。
- 评估记录可保存、可追溯、可复盘。
- 公共信用和资质核验可逐步从人工录入升级为联网核验。
- 后续可承接特批流程、审批记录、附件留痕和管理端。

## 开发节奏总原则

1. 每个 PR 只解决一个产品化问题。
2. `src/riskEngine.js` 始终是唯一风控计算入口，避免规则散落在 UI、数据库或接口层。
3. 数据库接入、联网核验、登录权限、审批流、管理端分开做。
4. 先上线一个可真实访问的内部 H5，再逐步加数据和流程。
5. 每轮完成后必须验证：规则测试、构建、手机视口、保存链路、控制台错误。

## 阶段路线

### Phase 0：原型基线冻结

目标：把当前 H5 原型作为 v0 基线固定下来。

范围：
- 初始化 Git 仓库。
- 建立 GitHub 远端。
- 保留当前 React + Vite 原型。
- 保留核心规则测试。
- 加 README，说明本地运行、业务规则、上线方式。
- 建立 GitHub Pages 或等价静态部署。

不做：
- 不接数据库。
- 不做登录。
- 不接联网核验。
- 不做审批流。

验收：
- 公开或半公开 HTTPS 地址可打开。
- 手机微信可访问。
- `npm test` 通过。
- `npm run build` 通过。

### Phase 1：数据访问层与本地/云端切换

目标：为数据库接入建立稳定边界，避免 UI 直接依赖数据库 SDK。

范围：
- 新增 `src/assessmentRepository.js`。
- 把当前 `localStorage` 读写从 `App.jsx` 中抽出。
- repository 提供统一方法：
  - `loadDraft`
  - `saveDraft`
  - `resetDraft`
  - `listRecords`
  - `saveRecord`
  - `loadRecord`
- 默认仍使用 localStorage。
- 预留 Supabase adapter，但不强行上线。

不做：
- 不改变 UI 主流程。
- 不改变评分规则。
- 不接公共信用联网核验。

验收：
- 当前功能体验不回退。
- 保存记录、刷新恢复、历史载入正常。
- repository 有单元测试。

### Phase 2：数据库接入 MVP

目标：让评估记录从单机 localStorage 进入云端数据库。

范围：
- 建 `assessment_records` 表。
- 保存评估快照：form snapshot + result snapshot。
- 列出最近记录。
- 载入记录。
- 明确匿名访问策略。

关键决策：
- 如果短期不做登录，建议只允许新增记录，读取范围必须受控。
- 更稳妥的方案是通过 Edge Function 代写入，不把敏感写权限暴露给前端。

不做：
- 不做复杂机构主数据。
- 不做审批流。
- 不做第三方核验。

验收：
- 换手机打开同一个 H5 可以看到云端记录。
- 刷新后记录仍存在。
- 不泄露 service role key。

### Phase 3：产品级信息架构

目标：从单页评估工具升级为清晰的业务应用。

范围：
- 拆分页面/视图：
  - 新建评估
  - 历史记录
  - 评估详情
  - 草稿恢复
- 增加状态：
  - 草稿
  - 已评估
  - 需特批
  - 已归档
- 增加详情页，用于复盘一次评估的输入、结果和原因。

不做：
- 不做多人权限。
- 不做审批流程。

验收：
- 历史记录不是简单列表，而是可进入详情复盘。
- 业务人员能回答“为什么当时给了这个判断”。

### Phase 4：联网核验与人工核验留痕

目标：把“核验页”从占位模块升级为可追溯的核验工作台。

范围：
- 建 `verification_logs` 表。
- 保存查询关键词、查询渠道、人工备注、核验状态。
- 支持上传或记录截图链接。
- 把核验结果映射回现有输入字段：
  - `publicCreditStatus`
  - `dishonestyHit`
  - `seriousIllegalHit`
  - `majorMedicalPenalty`
- 后续再接第三方或公开接口。

不做：
- 不直接改评分逻辑。
- 不把第三方接口失败当作自动拒绝。

验收：
- 每次公共信用判断都有来源记录。
- 评估结果能追溯到核验依据。

### Phase 5：特批流程

目标：把“需特批”从标签升级为流程。

范围：
- 建 `approval_requests` 表。
- 保存系统建议账期/额度、业务申请账期/额度、超规则原因、审批备注。
- 支持审批状态：
  - 待提交
  - 待审批
  - 已通过
  - 已拒绝
  - 需补充资料
- 支持审批意见留痕。

不做：
- 不一开始做复杂工作流引擎。

验收：
- 每个特批都有明确原因、风险提示和审批结果。

### Phase 6：登录权限与内部发布

目标：让系统适合真实内部使用。

范围：
- 登录。
- 用户角色：
  - 业务人员
  - 风控审核
  - 管理员
- RLS / 后端权限策略。
- 按角色控制读取和审批。

验收：
- 非授权人员不能读取全部记录。
- 普通业务不能篡改审批结果。

### Phase 7：管理端与报表

目标：支持经营复盘和风控策略迭代。

范围：
- 记录筛选。
- 风险等级分布。
- 特批率。
- 红线触发原因 Top N。
- 平均建议额度、平均申请额度。
- 机构维度历史变化。

验收：
- 可以用系统数据回答“哪些机构风险更高、规则是否过严或过松”。

## 推荐 PR 顺序

### PR 1：项目基线与在线静态部署

内容：
- 初始化 Git。
- 推 GitHub。
- 配 GitHub Pages / Actions。
- 加 README。
- 更新二维码为线上地址。

价值：
- 先让所有人能访问同一个版本。

### PR 2：数据访问层抽象

内容：
- 新增 repository。
- UI 不再直接读写 localStorage。
- 增加 repository 测试。

价值：
- 为数据库接入铺路。

### PR 3：数据库保存评估记录

内容：
- 接 Supabase 或等价数据库。
- 保存和读取 `assessment_records`。
- 保留 localStorage fallback。

价值：
- 从 demo 变成有数据沉淀的工具。

### PR 4：评估详情与历史记录产品化

内容：
- 历史列表进入详情页。
- 展示输入快照、结果快照、原因标签。

价值：
- 支持复盘和追责。

### PR 5：人工核验留痕

内容：
- `verification_logs`。
- 保存查询关键词、备注、截图引用、核验状态。

价值：
- 公共信用判断有依据。

### PR 6：特批流程 MVP

内容：
- 特批申请记录。
- 审批状态。
- 审批备注。

价值：
- 系统开始承接真实业务流程。

### PR 7：登录与权限

内容：
- 登录。
- 角色权限。
- RLS 或后端访问控制。

价值：
- 进入真实内部使用门槛。

## 上线节奏建议

### 第 1 个上线版本：V0.1 共享原型版

时间目标：最快 0.5-1 天。

目标：
- 有 HTTPS 地址。
- 微信扫码可访问。
- 仍然本地保存。

适用场景：
- 给业务同事看流程。
- 收集字段、规则、UI 反馈。

### 第 2 个上线版本：V0.2 云端记录版

时间目标：1-2 天。

目标：
- 数据库保存评估记录。
- 多设备可查看记录。
- 基础数据安全说明清楚。

适用场景：
- 小范围真实试用。

### 第 3 个上线版本：V0.3 可复盘版

时间目标：2-4 天。

目标：
- 历史详情。
- 评估快照。
- 核验备注。
- 原因标签可追溯。

适用场景：
- 真实业务试运行。

### 第 4 个上线版本：V1.0 内部流程版

时间目标：1-2 周，取决于登录和审批复杂度。

目标：
- 登录权限。
- 特批流程。
- 审批留痕。
- 管理端基础报表。

适用场景：
- 内部正式使用。

## 不能停留在 demo 的判断标准

以下任一项未满足，都还只是 demo：

- 没有统一线上地址。
- 数据只存在当前手机或当前浏览器。
- 不能追溯一次评估当时的输入、结果和原因。
- 特批只显示标签，没有流程和留痕。
- 公共信用判断没有查询依据。
- 没有权限边界。
- 没有测试守住风控规则。
- 没有版本发布记录。

## 每轮开发完成的固定复盘问题

1. 这一轮是否让系统更接近真实业务使用？
2. 是否保持 `riskEngine` 为唯一规则入口？
3. 是否有测试覆盖新增行为？
4. 是否能在手机端顺畅完成主流程？
5. 是否减少了后续数据库/权限/核验接入的复杂度？
6. 是否避免把临时 demo 写法固化进产品？

## 下一轮推荐提示词

推荐下一轮先做 PR 1：

```text
请按产品化路线图执行 PR 1：项目基线与在线静态部署。

要求：
1. 不改风控规则。
2. 不接数据库。
3. 初始化 Git 仓库并准备 GitHub 远端发布。
4. 使用 GitHub Pages 或最轻量稳定的静态托管方式部署当前 Vite H5。
5. 增加 README，说明产品目标、本地运行、测试、部署方式、当前限制。
6. 更新二维码为线上访问地址。
7. 完成后运行 npm test、npm run build，并用手机视口验证页面无横向滚动和控制台错误。
8. 在最终说明里明确：线上 URL、主要文件、验证结果、下一步 PR 建议。
```

如果你希望先接数据库，则下一轮提示词改为：

```text
请按产品化路线图执行 PR 2：数据访问层抽象。

要求：
1. 不直接接数据库，只先抽象 repository。
2. 保留 localStorage 行为不变。
3. App.jsx 不再直接操作 localStorage。
4. 新增 assessmentRepository，并为草稿、保存记录、历史列表、载入记录补测试。
5. 不改 riskEngine 规则口径。
6. 完成后运行 npm test、npm run build，并验证手机端保存/刷新/载入历史记录正常。
```
